Build a complete production-ready mobile app called NutriScan AI for an Indian audience.

The app is a packaged-food scanner that identifies a product from barcode first, uses OCR only as fallback, analyzes the nutrition label and ingredient list, personalizes warnings based on the user’s health profile, and returns a transparent health score out of 10.

IMPORTANT PRODUCT DIRECTION
- Build for India first, not as a generic US food app.
- Remove any standalone AI chat assistant entirely.
- AI must only be used inside product analysis, ingredient explanation, verdict generation, and personalization wording.
- The app must be fast, cheap to run, reliable on low-end Android phones, and usable on weak internet.
- The core scanner and result experience must be excellent before anything else.
- The app must feel trustworthy, not like a gimmick.
- Keep the health score explainable, transparent, and clearly broken down into reasons.
- Never pretend a low-confidence OCR result is accurate.
- Never show a result without showing confidence and data completeness.

PRIMARY USER GOAL
A user should be able to open the app, scan a packaged food item, and instantly understand:
1. what the product is,
2. what ingredients are inside it,
3. whether it is okay for their personal health profile,
4. what makes the score high or low,
5. whether the app is confident about the result.

CORE SCAN FLOW
The scan system must work in this order:
1. Barcode scan is always the first and fastest path.
2. If barcode resolves a product, use the resolved product data immediately.
3. If barcode fails, do OCR on the label.
4. If OCR succeeds, parse the text and generate structured product data.
5. If OCR still fails, allow manual search by name, brand, barcode, SKU, or reference code.
6. If no result is found, allow the user to submit a missing product.
7. If the app is uncertain, show a clear low-confidence state.
8. Never block the user with a dead end.

BARCODE SCANNING REQUIREMENTS
Use a free open-source barcode scanning solution.
Preferred approach:
- ZXing-based scanner or a ZXing-compatible open-source implementation.
- Support EAN-13, UPC-A, UPC-E, GTIN, QR where relevant.
- The barcode scanner must work quickly on cheap Android devices.
- Must include:
  - autofocus
  - torch toggle
  - blur warning
  - move closer guidance
  - retake prompt
  - scan frame overlay
  - vibration or subtle feedback on success
- The scanner must open instantly from the home screen and from every major page.
- Barcode scan should be the default mode because it is faster, cheaper, and more reliable than OCR.

OCR FALLBACK REQUIREMENTS
OCR is fallback only, not the main path.
Use OCR to read:
- ingredients
- nutrition facts panel
- product name
- brand
- allergen warnings
- serving size
- “contains” or “may contain” statements

OCR pipeline:
1. Capture image
2. Preprocess image
3. Crop likely label region if possible
4. Increase contrast and sharpen text
5. Run OCR
6. Clean OCR output
7. Normalize punctuation and spacing
8. Fix common OCR mistakes
9. Parse structured fields
10. Pass the structured output to the scoring engine and LLM explanation generator

OCR design rules:
- Handle mixed English and Hindi as best as possible.
- Handle curved packaging and noisy photos gracefully.
- If OCR quality is bad, show “text unclear” rather than inventing data.
- Give users a retake button and manual fallback.
- OCR should be fast enough to feel like a fallback, not a long waiting state.

OCR TECHNOLOGY CHOICE
Use the most practical low-cost option for the target device.
Recommended:
- Google ML Kit text recognition for on-device OCR
- Keep a backup option such as Tesseract if needed for development or edge cases
- Do not depend on expensive cloud OCR for the main flow unless absolutely necessary

SCORING SYSTEM: HEALTH SCORE OUT OF 10
The score must be transparent and explainable.

Use a weighted model that considers:
- calories / energy density
- added sugar
- sodium
- saturated fat
- trans fat
- fiber
- protein quality
- ingredient quality
- additive risk
- level of processing
- completeness of data
- personalization adjustment
- category-specific logic

Basic formula:
finalScore = clamp(0, 10, baseNutritionScore + ingredientQualityScore + additiveScore + personalizationAdjustment - missingDataPenalty - ultraProcessedPenalty)

The final score must be rounded to one decimal place.

The score must not pretend to be medically exact.
It is a practical consumer guidance score.

SCORING PRINCIPLES
1. High sugar should lower the score, especially for drinks and sweets.
2. High sodium should lower the score, especially for snacks, instant foods, and condiments.
3. Trans fat should be penalized heavily.
4. Saturated fat should be penalized moderately.
5. Fiber should help the score.
6. Protein should help the score, but should not hide bad ingredients.
7. Long artificial ingredient lists should reduce the score.
8. Additives should be classified clearly.
9. Personalized health conditions should modify relevance.
10. Missing or uncertain data should reduce confidence and slightly reduce trust in the score.
11. Category matters; a snack and a beverage cannot use the exact same threshold logic.

MODEL FALLBACK LOGIC
Implement model fallback:
1. Try primary model
2. If it fails, retry once
3. If it still fails, switch to fallback model
4. If fallback fails, use a rule-based generated summary from the scoring engine
5. Never fail the whole scan because the LLM failed

The fallback must be configurable from env.
The model name, API key, base URL, temperature, top-p, and max tokens must all be configurable.
